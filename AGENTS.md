## 核心编码实践

### 降低认知负荷

“代码首先是给人看的，其次才是给机器执行的。”

* **代码即文档**：好代码应该不需要注释就能理解。代码本身已经说明了“What”和“How”。注释只留给业务背景、晦涩逻辑的解释或特殊备注。
* **命名即注释**：用业务语言命名变量和函数，宁可名字长一点，也要消除歧义（例如用 `getUserAccountStatus` 代替 `checkStatus`）。
* **单一职责**：一个函数/类只做一件事。一个文件只负责一种职责。

### 童子军法则

“离开营地时，要让它比你来时更干净。” 

* **持续重构**：重构不是一个独立的阶段，而是开发的日常呼吸。在添加新功能的同时，顺手调整周围结构不合理的代码。尤其在内部开发阶段，应该对注意到的缺陷大胆进行重构，防止技术债务不断积累。
* **顺手清理**：在修改某个文件时，如果看到无用的废弃变量、拼写错误的函数名或格式不整齐的代码，顺手修复它。
* **不留破窗**：即使是极小的坏味道（Bad Smell），也不要假装没看见。随时重构。

### 实用主义

* **KISS (Keep It Simple, Stupid)**：用最简单直接的方法解决问题，不要为了未来“可能”出现的需求过度抽象。
* **YAGNI (You Aren't Gonna Need It)**：只写当前真正需要的代码。预测未来往往是错的，提前设计的接口大多会成为累赘。

## 工具规则

### Shell

不要用 powershell -Command 前缀，直接裸命令执行！
不要使用 &&（必须用 Powershell 5.X 语法）
禁止使用 2>$null 吞掉错误消息

### Git

- 使用任何 git restore / reset 前必须和用户确认

### Ask Questions

所有需要用户确认或指示的操作，都使用 Ask Question 工具执行。

## 多 Agent 协作

本工程同时由多个 Agent 修改，总是假定有其他 Agent 在同时工作。不要处理你的改动范围之外的变化。

## 默认测试范围

为核心真实场景和容易出错的逻辑添加测试用例。

除非用户要求：
- 不要编写过于显而易见的、仅为提高 coverage 数字的测试用例。
- 不要执行复杂的集成测试，如打开浏览器截图等等。
